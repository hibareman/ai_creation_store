AI Store Creation Pipeline 1. Core Pipeline Steps 1. Store Description Submission The user submits a textual description of the store idea through the system interface. This description may include general information such as the type of products, the target audience, or the overall concept of the store. After submitting the request, the system authenticates the user and extracts the tenant_id from the authentication context to ensure proper data isolation between tenants. The request is then forwarded to the AI Processing Service , which is responsible for generating the initial store draft. 2. Input Assessment by the AI The system sends a generation request to the AI service containing the store description provided by the user. Within the same request, the AI analyzes the description to determine whether the provided information is sufficient to generate an initial store draft. The AI response may result in one of the following outcomes: If the information is sufficient , the AI directly generates the store draft. If the information is insufficient , the AI returns clarification questions to collect the missing information. 3. Clarification Session (MCQs) If the AI determines that the description is not sufficient, the system presents a set of Multiple-Choice Questions (MCQs) to the user. These questions are designed to collect the essential information required to generate the store configuration, such as: the product category the type of products offered preferred store layout or template The user selects the appropriate answers, and the responses are merged with the original store description. The updated input is then sent again to the same AI generation process. This clarification cycle continues until sufficient information is available for the AI to generate the store draft. 4. Store Draft Generation and Interactive Presentation At this stage, after the initial AI-generated store draft is presented, the user has four interaction options: Manual Edit : The user can directly modify any field (e.g., store name, description, product price). Approve &amp; Generate Final Store : The user confirms the draft as-is, which triggers store creation. Partial Field Regeneration Partial regeneration allows the user to request AI assistance to regenerate a specific field within the store draft while preserving the rest of the configuration. Unlike manual editing, partial regeneration represents a separate AI interaction pattern that triggers a new request to the AI service. When the user selects the regeneration option for a specific field, the system constructs a new AI request containing the following context: the current store metadata (e.g., store name, description, category) the full current store draft structure the specific field that should be regenerated constraints specifying which fields must remain unchanged The AI service is explicitly instructed to regenerate only the selected field while maintaining compatibility with the rest of the draft. After receiving the response, the system performs validation using the Store Draft Validation Schema to ensure that: the regenerated field respects the defined data type and constraints the generated value remains consistent with the store category and other related fields If the validation succeeds, the regenerated field replaces the previous value in the store draft. If validation fails, the system rejects the generated value and allows the user to retry regeneration or continue editing manually. Each partial regeneration request is executed within the same user session , ensuring that the AI receives the most recent version of the store draft and maintains continuity throughout the editing process. Full Draft Regeneration Full regeneration allows the user to request the AI to generate a completely new store draft , replacing all generated fields. This option is useful when the user is not satisfied with the initial AI output and prefers to receive a new alternative configuration. When full regeneration is requested, the system constructs a new AI request containing: the original store description provided by the user any clarification answers previously collected the current store category or constraints defined during the session The AI service then generates a new complete store draft , including: store metadata suggested products Once the new draft is generated, the system performs mandatory validation using the Store Draft Validation Schema to verify that all required fields, data types, and structural constraints are satisfied. If the generated draft passes validation, it replaces the previous draft and is presented to the user for further review. If validation fails, the system rejects the generated output and redirects the process to Partial Regeneration or the Fallback Flow . Like partial regeneration, full regeneration is executed within the same user session , ensuring that the AI generation process maintains the correct tenant context and session continuity. 5. User Confirmation and Store Creation After reviewing and optionally modifying the generated store draft, the user confirms the final store configuration. Once the confirmation is received, the system creates the store according to the approved configuration and stores the data in the database. All stored data is associated with the tenant_id to ensure proper tenant isolation. 7 . Fallback Handling To ensure system availability and prevent workflow interruption, the platform defines explicit fallback conditions and recovery mechanisms when the AI service fails or produces unusable responses. 6.1 Fallback Trigger Conditions The fallback mechanism is activated when any of the following conditions occur during interaction with the AI service: AI API Error: If the AI service returns an HTTP error response such as 500, 502, 503, or 429, the request is considered failed. Response Timeout : If the AI service does not respond within the defined time threshold (10 seconds), the request is considered timed out and is cancelled. Malformed or Unparseable Response : If the AI response cannot be parsed, contains invalid JSON structure, or misses critical structural elements required by the system. Validation Failure :If the generated store draft fails to pass the mandatory validation rules defined in the Store Draft Validation Schema , such as missing required fields or violating structural constraints. When any of these conditions occur, the system redirects the request to the fallback flow. 6.2 Fallback Flow When fallback is triggered, the platform provides alternative options that allow the user to continue the store creation process without relying on the AI service. These options include: • providing a pre-built store template containing default categories and placeholder products • allowing the user to manually configure the store structure through the interface • enabling the user to retry the AI generation process later These mechanisms ensure that the store creation process remains functional even when the AI service becomes temporarily unavailable. 2.
