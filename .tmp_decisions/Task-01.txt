الـ Pipeline المعتمد للتنفيذ 1) Submit Store Description يقوم المستخدم بإرسال وصف المتجر عبر الواجهة . ما يتم في هذه المرحلة : التحقق من هوية المستخدم استخراج tenant context من السياق الموثوق التأكد أن الطلب مرتبط بالمستخدم الصحيح وبنفس الـ tenant 2) Create Store Immediately as Draft بمجرد إرسال الوصف، يقوم النظام بإنشاء Store مباشرة داخل قاعدة البيانات بحالة : draft ما يتم في هذه المرحلة : إنشاء سجل متجر حقيقي ربطه بالمالك الصحيح ربطه بالـ tenant الصحيح عدم إنشاء أي بيانات نهائية أخرى بعد اعتماد store_id نفسه في جميع المراحل اللاحقة 3) Send Input to AI بعد إنشاء المتجر بحالة draft ، يتم إرسال الوصف إلى خدمة الـ AI. ما يتم في هذه المرحلة : بناء prompt من الوصف إرسال الطلب إلى مزود الـ AI استخدام نفس store_id استخدام نفس المالك استخدام نفس tenant context 4) AI Input Assessment يقوم الـ AI أولًا بتقييم ما إذا كان الوصف كافيًا أم لا . النتيجتان المحتملتان : الوصف كافٍ → ينتقل مباشرة إلى توليد draft JSON الوصف غير كافٍ → يعيد clarification questions 5) Clarification Session إذا كانت المعلومات غير كافية، يقوم النظام بعرض MCQs أو أسئلة توضيحية على المستخدم . ما يتم في هذه المرحلة : عرض الأسئلة التوضيحية استقبال إجابات المستخدم دمج الإجابات مع الوصف الأصلي إعادة إرسال المدخلات إلى نفس AI flow الحفاظ على : نفس store_id نفس المالك نفس tenant context نفس draft session القرار المعتمد : الحد الأعلى لجولات clarification: 1–2 جولات فقط 6) AI Draft Generation عندما تصبح المعلومات كافية، يقوم الـ AI بإرجاع draft JSON منظم . هذا الـ draft يحتوي على : store metadata branding theme config categories initial products القرار المعتمد : المنتجات الأولية إجبارية العدد في الـ MVP: 2–4 منتجات أولية 7) Mandatory Validation بعد استلام draft JSON ، يقوم النظام بإجراء validation إلزامي قبل اعتماده للعرض . ما يتم التحقق منه : وجود جميع الحقول المطلوبة صحة أنواع البيانات صحة القيم والقيود توافق theme_template أو theme_template_id سلامة الفئات سلامة المنتجات uniqueness والمنطق البنيوي العام إذا فشل validation: لا يتم اعتماد draft يتم التوجيه إلى : partial regeneration أو full regeneration أو fallback flow 8) Save Temporary Draft إذا نجح validation ، يتم حفظ draft JSON بشكل مؤقت خارج الجداول الأساسية . القرار المعتمد : الـ AI draft ليس المصدر الرسمي النهائي يتم حفظه مؤقتًا فقط الجداول الأساسية لا تُكتب في هذه المرحلة 9) Interactive Review يقوم النظام بعرض draft على المستخدم للمراجعة . خيارات المستخدم في هذه المرحلة : Manual Edit Partial Field Regeneration Full Draft Regeneration Approve / Confirm 10) Partial Regeneration إذا طلب المستخدم إعادة توليد حقل محدد فقط، يتم إرسال طلب جديد إلى الـ AI مع الحفاظ على بقية draft. ما يتم في هذه المرحلة : تحديد الحقل المطلوب إعادة توليده إرسال السياق الكامل الحالي فرض بقاء باقي الحقول كما هي التحقق من الناتج الجديد استبدال الحقل فقط إذا نجح validation القرار المعتمد : لا يتم إنشاء store جديد تتم العملية داخل نفس session على نفس store_id 11) Full Draft Regeneration إذا طلب المستخدم إعادة توليد draft كامل، يتم إنشاء draft جديد كامل بدل السابق . ما يتم في هذه المرحلة : استخدام الوصف الأصلي استخدام clarification answers السابقة الحفاظ على نفس المتجر ونفس السياق استبدال draft المؤقت السابق إذا نجح validation 12) Fallback Flow إذا فشل الـ AI أو كان الرد غير صالح أو انتهت المهلة، ينتقل النظام إلى fallback. حالات تشغيل fallback: AI API error timeout malformed response unparseable JSON validation failure بدائل fallback: prebuilt store template manual configuration retry later 13) Confirm / Apply عند موافقة المستخدم على النتيجة النهائية، يتم تطبيق القيم المعتمدة على الجداول الحقيقية . ما يتم في هذه المرحلة : لا يتم إنشاء Store جديد يتم استخراج القيم من draft JSON يتم تطبيقها على : Store StoreThemeConfig Category Product initial records القرار المعتمد : هذه هي اللحظة الوحيدة التي تصبح فيها البيانات المولدة جزءًا رسميًا من النظام 14) Final Store Status Transition بعد نجاح confirm/apply ، يتم تحديث حالة المتجر وفق الـ lifecycle المعتمد : draft → setup → active التفسير : draft : المتجر أُنشئ وما زال تحت التوليد والمراجعة setup : تم اعتماد draft وتطبيقه لكن المتجر ما زال في مرحلة الإعداد active : المتجر أصبح جاهزًا وفعّالًا بالكامل القرارات المعتمدة نهائيًا يتم إنشاء Store مباشرة عند أول submit بحالة draft . لا يتم إنشاء Store جديد عند confirm. الـ AI يعيد draft JSON فقط . draft JSON ليس المصدر الرسمي النهائي . المصدر الرسمي النهائي بعد confirm هو : Store StoreThemeConfig Category Product initial records جميع عمليات clarification و regeneration تتم على : نفس store_id نفس المالك نفس tenant context نفس draft session theme_template أو theme_template_id هو الحقل الصحيح في قسم الـ theme. clarification محدود إلى 1–2 جولات . المنتجات الأولية إجبارية في الـ MVP بعدد 2–4 . lifecycle المعتمد لحالة المتجر : draft → setup → active نتيجة المهمة الأولى تم حسم : التدفق الرسمي الكامل توقيت إنشاء المتجر توقيت تطبيق البيانات دور الـ AI كمولد draft فقط دور confirm كمرحلة apply حدود clarification و regeneration إلزامية validation fallback flow lifecycle النهائي لحالة المتجر وبذلك تعتبر المهمة الأولى منجزة ومثبتة تصميميًا .
