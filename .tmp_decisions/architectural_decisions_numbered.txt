   1: Architectural Decisions for the AI-Assisted Store Creation System
   2: 1. Store Row Creation Timing
   3: Possible Implementations:
   4: Create the row immediately as Draft:The store record is created in the database with draft status as soon as the user submits the prompt, providing a store_id to link all related data instantly.
   5: Create the row only after final confirmation:The store record is created in the database only after the user confirms the final draft; all data remains temporarily in memory or session until confirmation.
   6: Chosen Decision:Create the store row immediately as Draft when the user submits the prompt.
   7: Justification:Having the store_id from the start simplifies linking all store components (pages, sections, settings) during AI generation and interactive editing. It reduces complexity and ensures smooth data management. The alternative would complicate partial edits and session data handling.
   8: 2. AI Draft Storage Location
   9: Possible Implementations:
  10: Django Session:Stores data within the user session. Suitable for small data but limited for large JSON structures and complex drafts.
  11: Redis with TTL:Stores data temporarily in memory with automatic expiration, ideal for large drafts, partial regeneration, and interactive edits.
  12: In-Memory Python Objects:Stores data in the application's memory. Fast but unreliable, as data is lost on server restart or multi-server setups.
  13: Chosen Decision:Store the AI draft in Redis with TTL.
  14: Justification:Redis provides fast, reliable, and temporary storage for AI drafts, supporting partial regeneration and interactive editing while automatically cleaning up expired drafts. Django sessions are limited for large or complex data, and in-memory storage is not reliable for production systems.
  15: 3. Lightweight AI Audit Log
  16: Possible Implementations:
  17: Add a lightweight log:Stores minimal information per AI operation (user_id, prompt summary, status, timestamp) without retaining full AI request/response.
  18: No log:No information is stored, which reduces complexity but limits error tracking and system monitoring.
  19: Chosen Decision:Implement a Lightweight AI Audit Log.
  20: Justification:This log enables system monitoring, debugging, and tracking AI operations without storing large datasets, keeping the database simple and manageable. Without it, troubleshooting AI failures or tracking usage would be difficult, especially in an interactive, AI-driven environment.
